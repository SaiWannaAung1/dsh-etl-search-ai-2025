### ðŸ—ï¸ File Roles in DshETL

| Folder | Responsibility | Extension |
| --- | --- | --- |
| **models/** | Definitions of your C# DTOs. | `.ts` |
| **services/** | Logic for calling your C# `/api` endpoints. | `.ts` |
| **viewmodels/** | State management using **Svelte Runes**. | `.svelte.ts`* |
| **routes/** | Presentation and binding to the ViewModels. | `.svelte` |

> **Note on `.svelte.ts`:** In Svelte 5, files containing runes like `$state` or `$derived` outside of a `.svelte` file must use the `.svelte.ts` extension so the compiler knows to process them.

### ðŸš€ Why this structure works for you

1. **Logical Grouping:** Your `SearchService.test.ts` will sit right next to the service, keeping logic and tests close.
2. **Scalability:** If you add a new feature (e.g., "User Profile"), you simply add a file in each folder (Model, Service, VM, Route).
3. **Clean Views:** Your `+page.svelte` files will be very short, usually just importing the VM and looping through results, making them easy to read and style with Tailwind.
