To implement this specific dependency chain in GitHub Actions, we use **Jobs** and the `needs` keyword. This ensures that the frontend build only starts if the backend is stable, and tests only run against a verified build.

### üìÇ File: `.github/workflows/frontend-pipeline.yml`

```yaml
name: Frontend CI Pipeline

on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main ]

jobs:
  # 1. Backend Integration Test (The Gatekeeper)
  backend-integration:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Setup .NET
        uses: actions/setup-dotnet@v4
        with:
          dotnet-version: '10.x'
      - name: Run Backend Integration Tests
        run: dotnet test --filter Category=Integration

  # 2. Frontend Build (Depends on Backend Success)
  frontend-build:
    needs: backend-integration
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: ./frontend
    steps:
      - uses: actions/checkout@v4
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
          cache-dependency-path: './frontend/package-lock.json'
      
      - name: Install dependencies
        run: npm ci
        
      - name: Build Frontend
        run: npm run build
        
      # Upload the build folder so the next jobs can use it
      - name: Upload Build Artifact
        uses: actions/upload-artifact@v4
        with:
          name: frontend-dist
          path: frontend/dist/

  # 3. Frontend Tests (Depends on Build Success)
  frontend-testing:
    needs: frontend-build
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: ./frontend
    steps:
      - uses: actions/checkout@v4
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          
      - name: Download Build Artifact
        uses: actions/download-artifact@v4
        with:
          name: frontend-dist
          path: frontend/dist/

      - name: Install dependencies
        run: npm ci

      - name: Run Vitest (Unit & Integration)
        run: npm run test:unit
        
      - name: Run Playwright (E2E Integration)
        run: |
          npx playwright install --with-deps
          npm run test:e2e

```

---

### üîç Explanation of the Pipeline Logic

#### **The Dependency Chain**

1. **`backend-integration`**: This is your "Base Layer." If the C# API logic is broken, there is no point in building the frontend.
2. **`frontend-build`**: Uses `needs: backend-integration`. It compiles the Svelte 5 code. We use `upload-artifact` to save the production-ready code.
3. **`frontend-testing`**: Uses `needs: frontend-build`. This job downloads the compiled code.
* **Unit/Integration Tests:** Runs Vitest against your ViewModels and Components.
* **E2E Integration:** Runs Playwright to ensure the "View" correctly navigates and interacts with the build.



---

### üõ†Ô∏è Why we use `upload-artifact`?

In GitHub Actions, every **Job** runs on a brand new, clean Virtual Machine.

* When the `frontend-build` job finishes, its files are deleted.
* By "Uploading" the build, we pass the compiled code to the `frontend-testing` machine so it doesn't have to rebuild the project, saving time and ensuring we are testing the **exact same files** that will go to production.
